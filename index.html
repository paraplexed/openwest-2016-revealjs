Has anyone gotten in a debate about a test type ? Example: what is 'regression testing' : Wikipedia definition: 'Regression testing is a type of software testing that verifies that software previously developed and tested still performs correctly after it was changed or interfaced with other software. Changes may include software enhancements, patches, configuration changes, etc'

How Google defines test types

"Instead of distinguishing between code, integration and system testing, Google uses the language of small, medium and large tests emphasizing scope over form"

Small tests

"Small Tests are mostly (but not always) automated and exercise the code within a single function or module"

Medium tests

"Medium Tests can be automated or manual and involve two or more features and specifically cover the interaction between those features"

Large tests

"Large Tests cover three or more (usually more) features and represent real user scenarios to the extent possible"

Benefits of defining your own test types

  • Team decides how the pipeline works and when to run tests more generally
  • Less one off cases of 'this one test should be run differently than the others'
  • Each type definition is unique to your team/company

Downsides of defining your own test types

  • Will take iterations to get right
  • What happens when a test doesn't fit into your definitions?

What does CI/CD look like?

Snap-ci

Example

Advantages

  • Easy to use UI
  • Docker beta
  • Good visual of pipeline
  • Robust

Disadvantages

  • Painful debugging
  • Slow UI
  • Everything done in the UI

Wercker

Example

Advantages

  • Easy to use UI
  • Good visual of pipeline
  • Built on Docker
  • Most everything done in yml file

Disadvantages

  • Still in Beta
  • Required to use their file format

Travis-ci

Example

Advantages

  • Everything done in a yml file
  • Tried and tested with years of open source

Disadvantages

  • Not made for CD
  • No pipeline view
  • Required to use their file format

Running tests outside of a pipeline

Small tests

  • Pre-commit or pre-push git hooks
  • Simple jenkins job polling repo

Medium tests

  • Pre-commit or pre-push git hooks
  • Simple jenkins job polling repo
  • Chain after 'deploy to environment' job (may require centralized medium test repo)

Large tests

  • Chain after 'deploy to environment' job
  • Scheduled

Handling failures

Notifications

  • Chat/email notifications on failure
  • Stack trace or short error message in notification
  • Jira bug created on failure, assigned to who caused test failure

Responsibility for looking at failing tests

  • Randomly assign someone on the team
  • Rotating goalie
  • First person into the office looks at failing tests

Automated testing in Docker

Benefits

  • Always clean environment
  • Always reproducible failures
  • Portable (nature of docker containers/images)

Downsides

  • Environment may not be the same as prod (unless you are deploying docker containers to prod)
