Benefits of defining your own test types

  • Team decides how the pipeline works and when to run tests more generally
  • Less one off cases of 'this one test should be run differently than the others'
  • Each type definition is unique to your team/company

Downsides of defining your own test types

  • Will take iterations to get right
  • What happens when a test doesn't fit into your definitions?

Running tests outside of a pipeline

Small tests

  • Pre-commit or pre-push git hooks
  • Simple jenkins job polling repo

Medium tests

  • Pre-commit or pre-push git hooks
  • Simple jenkins job polling repo
  • Chain after 'deploy to environment' job (may require centralized medium test repo)

Large/Enormous tests

  • Chain after 'deploy to environment' job
  • Scheduled

Handling failures

Notifications

  • Chat/email notifications on failure
  • Stack trace or short error message in notification
  • Jira bug created on failure, assigned to who caused test failure

Responsibility for looking at failing tests

  • Randomly assign someone on the team
  • Rotating goalie
  • First person into the office looks at failing tests

Automated testing in Docker

Benefits

  • Always clean environment
  • Always reproducible failures
  • Portable (nature of docker containers/images)

Downsides

  • Environment may not be the same as prod (unless you are deploying docker containers to prod)
